Breach Report

Log vulnerability findings for tracking and remediation.

Security teams use intake forms like this to standardize how findings are documented. Each submission captures severity, affected assets, and status, creating an audit trail that flows into Admin Logs.

Severity LevelsLow → Critical

Classify findings by impact to prioritize remediation efforts across the engagement.

Tracking StatusNew → Remediated

Each finding moves through a lifecycle: New, In Review, Validated, and Remediated.

Submit Finding

Document a discovered vulnerability. All fields map to the planned MySQL vuln_logs table from the requirements document.

Report Workflow

Step 1

Document the finding

Fill in the client, severity, affected asset, and a clear summary with recommended next steps.

Step 2

Data is stored locally

The prototype saves entries to browser storage, simulating the SQL INSERT that the backend version will perform.

Step 3

Review in Admin Logs

Navigate to Admin Logs to see all submitted findings in a sortable table — the same view an engagement lead would use.
